Everything deterministic (process mining, KPI computation, RAG status) lives in TypeScript and runs against the database.

End-to-end user flow

A consultant logs in, creates an engagement, scopes one or more processes, drafts an ideal model + data request, ingests CSVs, then drives the digital twin through to a quantified findings narrative.

Consultant flow

1
Sign in
NextAuth credentials provider. Role determines admin visibility (analyst vs admin).
2
Create or open an Engagement
Capture client profile: country/region (region auto-derived), institution type, AUM, employees, customers, core banking system. The engagement's country drives ValueCoefficient lookups in Stage 5.
3
Add Processes (one or many)
Pick a line of business (Retail / SME / Commercial / Wealth) and a process. Bootstrapped from the active CountryProcessActivation if available, else from the canonical ProcessTemplate. Snapshot of templateVersion recorded.
4
Process landing — three tiles
Tile 1: Process Model & Data Request · Tile 2: Process Discovery & Insights · Tile 3: APA Simulation & Value Recovery (locked).

Tile 1 — Process Model & Data Request

A
Edit the BPMN canvas
React Flow + Dagre layout. Drag/drop sub-processes, decisions, end states. State persisted to EngagementProcess.processMap.
B
Capability assessment
For each sub-process, mark digital / partial / manual. Persisted to processCapabilities. Drives the data-request scoping below.
C
Generate Data Request
Claude takes canvas + capability state, returns a MoSCoW-prioritised data request (must / should / could / won't), saved to dataRequest. Sample CSVs deterministically match the request — same item set every download.

Tile 2 — Process Discovery & Insights (5 inline stages)

1
Stage 1 — Upload
Drag CSVs into per-data-request slots. Auto-match by filename. Reset clears stale uploads. Per-file role auto-detection: case_id, activity, timestamp, actor.
2
Stage 2 — Build Activity Table
Heuristic + value-based column detection, parseTimestamp rejects years outside 1900-2100, chunked event inserts (CHUNK=2000), rawData purged after build. Result: rows in EventLog ready for mining.
3
Stage 3 — Process Explorer + Variants
React Flow graph with two sliders (min activity %, min path %) defaulting to clean view. Heat-map toggle (frequency / cycle time). Pink edges mark exception flows. Variant analysis lists top-N paths with frequency + median cycle time.
4
Stage 4 — KPI Cockpit
8 categories (time / volume / quality / outcome / cost / cx / workforce / compliance) × 3 sources (direct / inferred / assumed). RAG status vs benchmarks. Click any assumed metric to override per engagement; assumed metrics are stored as defaults in metricDefinitions and overridden per-engagement in metricOverrides.
5
Stage 5 — Findings (Value Leak)
Claude narrates findings grouped by cockpit category, with rank, severity, cases affected, annual value leak (computed from coefficients), root cause, recommendation, and recommended APA agent.

Admin flow (role: admin)

A
Admin home — three tiles

B
Process Repository → click a process
Header shows Active/Draft + version + line of business. Sections: Steps · Systems · Actors · Deviation Library · Sub-processes · Metric Definitions. Process Explorer Agent button (Claude) generates a full process from a name + description.

Activated Processes (links to canonical templates) + Value Coefficients table (CRUD). New country activation: pick canonical version → optionally customise map / sub-processes / metrics → activate.

AI vs Database vs Algorithm

Every feature is classified by where its data comes from. Claude is invoked only where structured generation or narrative synthesis is the actual product. Database rows answer everything that's a curated fact or persisted state. Algorithm covers deterministic computation over those rows.
Claude AI call to claude-sonnet-4-20250514
Database Postgres read/write via Prisma
Algorithm deterministic TS code

Tile 1 — Process Model & Data Request
BPMN canvas (drag/drop, save) Database EngagementProcess.processMap React Flow state serialised to Json on save. No AI.
Default process map for a new process Database ProcessTemplate.defaultProcessMap · CountryProcessActivation.customProcessMap Bootstrap cascade: country variant → canonical template.
Capability assessment (digital/partial/manual) Database EngagementProcess.processCapabilities Form input persisted to Json. No AI.
Sub-process rows shown in capability form Database ProcessTemplate.subProcesses Drives the capability rows the consultant assesses.
Generate Data Request (MoSCoW) Claude /api/engagements/[id]/data-request · DataRequestTemplate.items Claude takes the canvas + capability state and returns a MoSCoW-prioritised data request. Seeded from DataRequestTemplate when present.
Sample CSV download Algorithm /api/engagements/[id]/processes/[processId]/sample-csvs Deterministic generator that writes one CSV per DataRequestItem with the right columns. Bundled as a zip via JSZip.
Tile 2 — Stage 1 Upload
Schema inference (column → role) Claude /api/schema-inference · Upload.schemaInference Claude proposes column roles (case_id / activity / timestamp / actor / attribute). Result cached on the upload row.
Auto-match upload → data-request slot Algorithm Upload.dataRequestFileName Filename match against DataRequestItem.fileName; same-name uploads auto-replace the prior file.
Tile 2 — Stage 2 Build Activity Table
Column role binding (heuristic) Algorithm activityTable.ts:guessColumnByRole Header-name heuristic + value-shape sniffing. AI-inferred schema only used as a hint.
Timestamp parsing Algorithm activityTable.ts:parseTimestamp Rejects new Date("675") year-675 bug; only accepts dates in 1900-2100.
EventLog persistence Database EventLog Chunked inserts (CHUNK=2000). Upload.rawData nulled after build to keep storage flat.
Tile 2 — Stage 3 Process Explorer + Variants
Process graph (nodes, edges, frequencies, cycle time) Algorithm processGraph.ts · reads EventLog Group events by case → derive directly-follows graph → aggregate frequency & median cycle time per edge → dagre layout with happy-path edge weighting (10×).
Variant analysis (top-N + long tail) Algorithm variants.ts Hash each case's activity sequence, count, sort, bucket the long tail.
Pink exception edges Algorithm processGraph.ts + deviationLibrary.ts Edges flagged when target activity matches a deviation pattern step keyword.
Tile 2 — Stage 4 KPI Cockpit
Metric definitions (catalog) Database ProcessTemplate.metricDefinitions 8 categories × 3 sources, with thresholds, direction, unit, formula. Curated per process.
Direct metrics (cycle time, throughput, etc.) Algorithm cockpitMetrics.ts:computeDirect · reads EventLog Keyword-matched aggregations over the event store. No AI.
Inferred metrics (cost, FTE-driven) Algorithm + Database cockpitMetrics.ts:computeInferred + ValueCoefficient Formula evaluator that pulls FTE rates and other coefficients from the country/institution scope. Touch-time uses TOUCH_RATIO=0.18 banking estimate.
Assumed metrics (defaults) Database ProcessTemplate.metricDefinitions[].defaultValue Default value carried by the metric definition.
Per-engagement metric overrides Database EngagementProcess.metricOverrides Click an assumed metric → save number → PATCH /metric-overrides. Override flag rendered as a purple pill.
RAG status vs benchmarks Algorithm cockpitMetrics.ts:evaluateStatus Compare value to goodThreshold / poorThreshold with respect to direction (lower/higher is better).
Tile 2 — Stage 5 Findings (Value Leak)
Detect deviations from graph Algorithm findings.ts:detectDeviations Scan the discovered graph + variants for skip / loop / out-of-order / extra-step patterns.
Match deviations to library Algorithm findings.ts:matchToLibrary + DeviationPattern + deviationLibrary.ts Pattern key + step keyword + scope (process, country) match against the curated library to attach candidate reasons.
Findings narrative + recommendations Claude /api/engagements/[id]/processes/[processId]/findings · findings.ts:buildFindingsPrompt Claude takes detected deviations + matched reasons + cockpit + coefficients, returns ranked findings with narrative, root cause, value-leak breakdown, recommendation, and recommended APA agent.
Annual value leak quantification Algorithm + Database findings.ts + ValueCoefficient Cases-affected × per-case cost (FTE rate × touch hours, breach fines, lost revenue) using the engagement's country coefficients.
Admin — Process Repository
Process Explorer Agent (generate a full process) Claude /api/admin/generate-process Name + description in → steps, actors, systems, deviation patterns + reasons, metric definitions out. Upserts ProcessTemplate + DeviationPattern + legacy step/actor/system rows.
Process Templates list / detail / CRUD Database ProcessTemplate · /api/admin/process-templates Plain CRUD with versioning.
Metric Definitions admin Database ProcessTemplate.metricDefinitions Inline editor: category, source, unit, formula, thresholds, direction, required.
Deviation Patterns (per process) Database DeviationPattern · /api/admin/deviation-patterns Edit/delete patterns + individual reasons. 9 seeded patterns covering KYC, credit, fraud, document, decision/account ordering.
Admin — Country Repository
Activated processes per country Database CountryProcessActivation Country forks a canonical version; engagements with matching (country, processKey) bootstrap from this.
Value Coefficients (FTE rates, fines, etc.) Database ValueCoefficient Scoped by (country, institutionType="" sentinel, key, validFrom). Cascade: country+inst → country → fail closed.
Auth & misc
Login Database User · NextAuth credentials bcrypt password check; role determines admin visibility.
Engagement create / list / detail Database Engagement Plain CRUD.

Known bugs (deferred)

Bug
Sample CSV simulator: actors uniformly random per row
In sample-csvs/route.ts:99-100 the actor cell for every event is picked uniformly at random from a fixed 6-name pool (customer, front_office_agent, compliance_officer, underwriter, system, fraud_analyst) regardless of which activity the row represents. Result: every activity in the explorer shows ~17% across all 6 actors instead of one or two dominant actors. Fix: make the generator activity-aware (e.g. KYC → 60% system, 30% compliance_officer, 10% customer) and reflect realistic banking data quality (heavy SYSTEM / service-account share, occasional named users). Doesn't affect production data — only the demo / sample CSVs.
